# 관리자 페이지 접근 제한 — 설계

## 배경 / 목표

`2_관리자대시보드(admin).html`은 서버 없이 브라우저에서 동작하는 정적 파일이고, 현재는 `firebase.auth().signInAnonymously()`로 누구나 인증 없이 접속해 전체 지점의 오늘 현황과 최근 30일 소요시간 통계를 볼 수 있다.

목표는 단순한 UI 가림막이 아니라 **Firestore 데이터 자체를 보호**하는 것이다. 클라이언트 JS에 심어둔 PIN은 브라우저 개발자도구로 그대로 노출되거나 우회되어 실질적 보호가 되지 않으므로, 보호는 반드시 Firestore 서버 쪽에서 강제되는 **Firestore 보안 규칙**으로 구현한다. 규칙이 요청자를 구분하려면 진짜 인증된 신원이 필요하므로, 관리자 전용 로그인(Firebase 이메일/비밀번호 인증)을 도입한다.

`1_입력시스템(index).html`(기사 앱)은 이번 변경 범위에 포함되지 않으며 계속 익명 인증으로 동작한다.

## 결정한 접근 방식

Firebase Authentication(이메일/비밀번호) + Firestore 보안 규칙 조합. 새로운 백엔드 서버는 도입하지 않는다 — Firebase Auth는 Firestore와 마찬가지로 서버리스 관리형 기능이며, 프로젝트의 "서버 없음" 원칙([CLAUDE.md](../../../CLAUDE.md) 참고)을 유지한다.

검토했지만 채택하지 않은 대안: 클라이언트 JS에 PIN을 하드코딩해 UI만 가리는 방식 — 구현은 간단하지만 Firestore 데이터 자체는 보호되지 않아 목표를 충족하지 못한다.

## 1. 로그인 흐름 & UI (admin.html)

- `initFirebase()`가 무조건 `signInAnonymously()`를 호출하던 방식을 제거하고, `firebase.auth().onAuthStateChanged()` 리스너로 인증 상태를 관찰한다.
- 로그인 안 된 상태: `renderLogin()` 화면 표시 — 비밀번호 입력창 1개 + "로그인" 버튼만 노출. 이메일 입력은 없고, 코드 상수 `ADMIN_EMAIL`을 내부적으로 사용해 `firebase.auth().signInWithEmailAndPassword(ADMIN_EMAIL, 입력값)`을 호출한다.
- 로그인 성공: 기존 대시보드(`view === "dashboard"`)/통계(`view === "stats"`) 화면 구조는 그대로 두고, 진입 조건만 인증 상태에 의존하도록 바꾼다.
- Firebase Auth는 기본적으로 로그인 상태를 브라우저에 유지하므로(LOCAL persistence), 재방문 시 재로그인이 필요 없다. 대시보드 상단(topbar)에 "로그아웃" 버튼을 추가한다.
- `ADMIN_EMAIL`은 코드에 상수로 남지만 비밀이 아니라 계정 식별자일 뿐이다. 실제 비밀번호는 코드 어디에도 저장되지 않고 Firebase 콘솔에서 관리자 계정을 만들 때만 설정되며, Firebase 서버가 검증한다.
- **스타일 재사용**: 새 CSS를 추가하지 않고 admin.html에 이미 있는 클래스를 그대로 쓴다. 로그인 화면은 `.form-card`/`.field`/`.save-btn` 조합으로 구성하고, 로그인 실패 메시지는 기존 `.err-banner`를 재사용한다. topbar는 "배송 관리자 대시보드" 타이틀만 남기고 그 아래 로그인 폼을 배치해 기존 화면 톤과 일치시킨다.

## 2. Firestore 보안 규칙 & 기사 앱과의 경계

- **쓰기(create/update)**: 지금처럼 익명 사용자(기사 앱)도 허용 — 운송시작/완료 등록은 이번 범위 밖이며 기존 동작을 유지한다.
- **오늘 날짜 문서 읽기**: 익명 사용자도 허용 — 기사 앱의 "등록 목록" 표시, 관리자 대시보드의 오늘 현황 조회 둘 다 필요로 한다.
- **오늘이 아닌 날짜 범위 조회(30일치 통계 쿼리)**: `request.auth.token.email == ADMIN_EMAIL`인 경우에만 허용 — 이번 설계의 핵심 보호 대상.
- 정확한 Firestore 규칙 문법은 이 설계 문서에서 확정하지 않고 구현 단계에서 작성한다.
- `firestore.rules` 파일을 리포에 참고용으로 추가하여, Firebase 콘솔의 규칙 탭에 그대로 붙여넣을 수 있게 문서화한다. 이 리포에는 배포 스크립트/CLI 설정이 없으므로, `firebaseConfig` placeholder를 콘솔 값으로 교체하는 것과 마찬가지로 **콘솔에 수동으로 반영해야 한다**는 점을 CLAUDE.md에도 남긴다.

## 3. 에러 처리

- 로그인 실패(`auth/wrong-password`, `auth/invalid-credential` 등): 로그인 화면에 "비밀번호가 올바르지 않습니다" 에러 메시지 표시.
- 연결 실패(Firebase 초기화 실패 등): 기존 `loadError` 배너 패턴을 그대로 재사용한다.
- 로그아웃: `firebase.auth().signOut()` 호출 → 로그인 화면으로 복귀, 기존 구독(`unsubDashboard`, `unsubStats`) 해제.

## 4. 검증 방법 (수동)

이 프로젝트에는 테스트 스위트가 없으므로 다음을 수동으로 확인한다.

1. 올바른 비밀번호로 로그인 → 대시보드/통계 정상 진입
2. 틀린 비밀번호 → 에러 메시지 표시, 대시보드 진입 안 됨
3. 로그아웃 → 로그인 화면으로 복귀
4. 브라우저 개발자도구 콘솔에서 로그인 없이 직접 Firestore 30일치 range query 시도 → 거부되는지 확인 (규칙이 실제로 동작하는지의 핵심 검증)
5. 기사 앱(index.html)이 기존과 동일하게 로그인 없이 정상 동작하는지 확인

## 범위 밖 (이번 브레인스토밍에서 함께 검토했으나 제외)
