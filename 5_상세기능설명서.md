# 운송기사 배송 입력 시스템 — 상세 기능 설명서

## 1. 주요기능

### 기사 앱 (index.html)
- **2단계 등록 흐름**: 1단계에서 권역(수도권/지방)과 지점을 다중 선택하고, 2단계에서 선택 내역을 확인한 뒤 "운송시작"을 누르면 선택된 지점 전체가 동일 시각으로 일괄 등록됨
- **지방 전체선택**: 지방 지점은 "전체 선택" 버튼으로 한 번에 모두 선택 가능 (트럭 한 대가 여러 지방 지점을 함께 도는 구조를 반영)
- **동시 기사 중복 등록 방지**: 오늘 이미 다른 기사가 등록한 지점은 선택 화면에서 자동으로 비활성화되고 "등록됨"/"완료" 라벨 표시
- **운송완료 처리**: 확인창(되돌릴 수 없음을 안내)을 거쳐 완료 처리하며, 완료 즉시 목록에서 사라짐
- **지방 일괄 운송완료**: 진행중인 지방 지점이 2개 이상이면 "지방 N개 지점 허브 도착(일괄 운송완료)" 버튼이 나타나 한 번에 전체 완료 처리 가능
- **자정 자동 갱신**: 기기를 자정 넘게 켜둬도 새로고침 없이 날짜가 자동으로 갱신됨
- **테스트 모드**(`?test=1`): 실제 데이터베이스에 연결하지 않고 샘플 데이터로 모든 조작을 안전하게 미리 체험 가능

### 관리자 앱 (admin.html)
- **로그인 보호**: 이메일/비밀번호 인증을 통과해야 대시보드 접근 가능
- **실시간 대시보드**: 36개 전 지점(수도권 24 · 지방 12)을 대기/운송중/완료 3색으로 실시간 표시
- **지연 판정 배지**: 경과시간이 평균(수도권은 해당 지점, 지방은 지방 전체 평균) 대비 15분 이상 초과하면 "지연" 배지 표시
- **소요시간 통계**: 최근 30일 중 조회일을 선택해, 지점별 월평균 소요시간과 비교한 증감을 색상으로 표시

### 공통
- 양쪽 앱 상단에 교보문고 CI(심볼+워드마크) 표시
- PC·모바일 어디서든 브라우저로 접속 가능한 반응형 웹앱

## 2. 작동방식

### 아키텍처
서버를 별도로 두지 않는 완전 서버리스 구조입니다.

```
[기사 앱] ──┐                    ┌── [관리자 앱]
            │                    │
            ▼                    ▼
        Firestore (실시간 데이터베이스, deliveries 컬렉션)
            │
            ▼ (5분마다 REST API로 읽어감)
      Google Apps Script ──▶ Google 스프레드시트 (영구 백업)
```

- **Firebase Firestore**: 배송 기록이 저장되는 실시간 NoSQL 데이터베이스. 문서 하나가 "지점 하나의 오늘자 운송 1건"을 의미
- **실시간 동기화**: 두 앱 모두 Firestore의 `onSnapshot` 리스너를 구독하고 있어, 누군가 데이터를 등록/변경하면 새로고침 없이 화면이 즉시 갱신됨
- **Firebase Hosting**: 두 HTML 파일을 정적 파일로 배포하는 웹 서버 역할
- **Google Apps Script**: 5분마다 자동 실행되는 트리거가 Firestore REST API로 전체 데이터를 읽어와 Google 스프레드시트를 통째로 갱신 (소요시간 자동 계산 수식 포함)

### 인증 방식
- **기사 앱**: 별도 로그인 없이 "익명 인증(Anonymous Auth)"으로 자동 연결 — 누구나 바로 쓸 수 있되, Firestore 보안 규칙으로 오늘 날짜 데이터만 읽고 쓸 수 있게 제한
- **관리자 앱**: 이메일/비밀번호 로그인 필요, 로그인된 계정만 전체 기간 데이터 조회 가능 (30일 통계용)

### 데이터 흐름 (한 건의 배송이 기록되는 과정)
1. 기사가 지점을 선택하고 "운송시작"을 누르면 → Firestore에 새 문서 생성 (지점명, 권역, 날짜, 시작시각 기록)
2. 관리자 화면이 실시간으로 "운송중"으로 갱신됨
3. 기사가 "운송완료"를 누르면 → 같은 문서에 완료시각만 추가로 기록
4. 관리자 화면이 실시간으로 "완료"로 갱신되고, 기사 화면에서는 해당 카드가 사라짐
5. 5분 이내에 Apps Script가 이 문서를 읽어가 스프레드시트에 자동 반영

## 3. 사용절차

### 기사
1. 기사 앱 URL(`/driver`) 접속 → 자동으로 연결됨 (로그인 불필요)
2. 권역(수도권/지방) 선택
3. 오늘 운송할 지점을 하나 이상 선택 (지방은 "전체 선택"도 가능)
4. "다음" → 선택 내역 확인 → "운송시작" 클릭 → 등록 완료
5. 배송 진행 중에는 화면 하단 목록에서 진행 상황 확인 가능
6. 도착 후 "운송완료" 클릭 → 확인창에서 재확인 → 완료 처리 (지방은 여러 지점을 "일괄 운송완료"로 한 번에 처리 가능)

### 관리자
1. 관리자 앱 URL(`/admin`) 접속
2. 이메일/비밀번호로 로그인
3. 대시보드에서 36개 지점의 실시간 현황(대기/운송중/완료) 및 지연 여부 확인
4. 우측 상단 "소요시간 통계" 클릭 → 조회일 선택 → 지점별 월평균 대비 증감 확인

## 4. 입력/출력값

### 입력값 (기사가 직접 입력하는 것)
| 항목 | 설명 |
| --- | --- |
| 권역 | 수도권 / 지방 중 선택 |
| 지점명 | 선택한 권역 내 지점 (다중 선택 가능) |
| 운송시작/운송완료 클릭 | 버튼 클릭 시점의 시각이 자동 기록됨 (기사가 시각을 직접 타이핑하지 않음) |

### 시스템이 자동으로 생성하는 값
| 필드 | 설명 |
| --- | --- |
| `date` | 등록 시점의 날짜(YYYY-MM-DD), 조회 필터 기준 |
| `startAt` | 운송시작 클릭 시각 (Timestamp) |
| `endAt` | 운송완료 클릭 시각 (Timestamp, 완료 전엔 없음) |
| `createdAt` | 문서 생성 서버 시각 (정렬용) |

### 출력값 (관리자/기사 화면에 보이는 것)
| 화면 | 출력 내용 |
| --- | --- |
| 기사 앱 목록 | 진행중인 지점의 시작시각, 흐름 표시 |
| 관리자 대시보드 | 지점별 상태(대기/운송중/완료), 경과시간, 지연 여부, 전체 지점 수 대비 완료 비율 |
| 소요시간 통계 | 지점별 월평균 소요시간(분), 조회일 소요시간, 증감(분) |
| 스프레드시트 | 지점·권역·날짜·운송시작·운송완료·소요시간(자동 수식) 전체 이력 |

## 5. 주요특징

- **완전 서버리스**: 별도 서버 구축·운영 비용 없이 Firebase와 Google 계정만으로 동작
- **실시간성**: 등록/완료 내역이 새로고침 없이 관리자 화면에 즉시 반영됨
- **동시 사용 안전성**: 여러 기사가 동시에 같은 지점을 중복 등록하지 못하도록 원천 차단
- **지방 배송 특화 로직**: 지방은 개별 지점이 아닌 "중간 허브 도착" 기준으로 소요시간을 계산하고, 완료 처리도 트럭 단위로 일괄 처리 가능
- **오조작 방지**: 운송완료는 되돌릴 수 없다는 점을 명시한 확인창을 거침
- **안전한 테스트 환경**: `?test=1`로 실제 데이터에 영향 없이 기능을 미리 체험 가능
- **이중 백업 구조**: 실시간 데이터(Firestore)와 5분 주기 영구 백업(스프레드시트)을 병행해 데이터 유실 위험을 낮춤
- **반응형 디자인**: PC·모바일 구분 없이 하나의 코드로 동작
